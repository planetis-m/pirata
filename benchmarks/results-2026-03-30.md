# Pirata Microbench Results

Environment:
- Date: 2026-03-30
- Compiler: Nim 2.3.1
- Target: Linux amd64
- Command: `nim c -d:danger -r benchmarks/microbench.nim`
- Runtime config from Nim: `mm: orc`, `threads: on`, `opt: speed`

The benchmark suite lives in [`benchmarks/microbench.nim`](./microbench.nim) and focuses on slot-table lookups plus the public ECS hot paths:
- `slot_table_lookup`
- `spawn_destroy_cycle`
- `add_fetch_remove`
- `tag_toggle`
- `query_update`
- `destroy_payload_dense`

## Baseline

Median ns/op before optimization work:

| benchmark | median ns/op |
| --- | ---: |
| `slot_table_lookup` | 1.93 |
| `spawn_destroy_cycle` | 2.64 |
| `add_fetch_remove` | 3.15 |
| `tag_toggle` | 2.57 |
| `query_update` | 5.94 |
| `destroy_payload_dense` | 19.58 |

## Experiments

### 1. Reuse the resolved signature slot

Change:
- added `SlotTable.lookupIndex`
- reused the resolved signature slot inside `contains`, `has`, `add`, `fetch`, `remove`, and `destroy`
- stopped re-validating the same entity handle multiple times inside one public ECS call

Impact from the first post-change run:

| benchmark | before | after | delta |
| --- | ---: | ---: | ---: |
| `slot_table_lookup` | 1.93 | 1.66 | -14.0% |
| `spawn_destroy_cycle` | 2.64 | 2.26 | -14.4% |
| `add_fetch_remove` | 3.15 | 2.41 | -23.5% |
| `tag_toggle` | 2.57 | 1.53 | -40.5% |
| `query_update` | 5.94 | 4.69 | -21.0% |
| `destroy_payload_dense` | 19.58 | 16.65 | -15.0% |

Decision:
- kept

### 2. Walk `query` directly instead of stacking iterators

Change:
- replaced `for entry in world.signatures.pairs` with a direct indexed loop over signature storage

Impact:
- `query_update` regressed from `4.69` to `4.81` median ns/op
- `add_fetch_remove` regressed from `2.41` to `2.52` median ns/op

Decision:
- rejected

### 3. Use `zeroMem` for copyable component clears

Change:
- replaced `typedData[T](data)[slot] = default(T)` with `zeroMem(...)` inside `clearColumnSlot` for `supportsCopyMem(T)`

Impact from the first run after the change:

| benchmark | before | after | delta |
| --- | ---: | ---: | ---: |
| `add_fetch_remove` | 2.41 | 2.41 | 0.0% |
| `query_update` | 4.69 | 4.58 | -2.3% |
| `destroy_payload_dense` | 16.65 | 15.40 | -7.5% |

Decision:
- kept

### 4. Skip the second destroy-time validation

Change:
- added `SlotTable.delAt`
- `world.destroy` now validates once, clears payload columns, then unlinks the entity directly by slot index

Impact from the first run after the change:

| benchmark | before | after | delta |
| --- | ---: | ---: | ---: |
| `spawn_destroy_cycle` | 2.42 | 2.13 | -12.0% |
| `tag_toggle` | 1.56 | 1.53 | -1.9% |
| `query_update` | 4.58 | 4.61 | +0.7% |

Decision:
- kept

### 5. Remove hot-path runtime validation by default

Change:
- the default API now trusts the world after setup instead of validating every entity on `add`, `fetch`, `remove`, `has`, and `destroy`
- registration-time checks stay in place

Impact from the first unchecked run:

| benchmark | before | after | delta |
| --- | ---: | ---: | ---: |
| `slot_table_lookup` | 1.65 | 1.66 | +0.6% |
| `spawn_destroy_cycle` | 2.13 | 2.00 | -6.1% |
| `add_fetch_remove` | 2.41 | 1.09 | -54.8% |
| `tag_toggle` | 1.60 | 0.71 | -55.6% |
| `query_update` | 5.00 | 2.30 | -54.0% |
| `destroy_payload_dense` | 16.86 | 9.80 | -41.9% |

Decision:
- kept

## Final Take

Final median ns/op from the last verification run:

| benchmark | final median ns/op | total improvement vs baseline |
| --- | ---: | ---: |
| `slot_table_lookup` | 1.66 | -14.0% |
| `spawn_destroy_cycle` | 2.00 | -24.2% |
| `add_fetch_remove` | 1.09 | -65.4% |
| `tag_toggle` | 0.71 | -72.4% |
| `query_update` | 2.30 | -61.3% |
| `destroy_payload_dense` | 9.80 | -49.9% |

The dominant remaining cost was still runtime validation in the hot API. Once that was removed from the default path, the public ECS operations roughly halved again on the mutation and query-heavy cases.

Remaining opportunities worth exploring later:
- a query API that yields borrowed component references directly instead of re-looking-up each entity during `fetch`
- a denser entity handle layout with more version bits if long-running reuse patterns matter more than the current compact handle
- specialized batch APIs for `add/remove/fetch` if the target workload spends most of its time in bulk transforms instead of ad-hoc entity edits
- better benchmark isolation for the dense-destroy case, which is still the noisiest scenario in this suite
